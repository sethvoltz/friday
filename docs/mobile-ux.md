# Mobile UX

Friday is mobile-first, not mobile-responsive-as-an-afterthought. Every page is fully usable on a phone, including chat with attachments, agent inspection, and dashboard tables.

## Layout primitives

- **Fixed header** on all viewports. No bottom tab bar.
- **The document scrolls on every route EXCEPT chat.** Most pages are an inert in-flow body that scrolls the document/window. The **chat route is the exception (ADR-041):** a `position: fixed` column (`.chat-viewport`) sized to the visual viewport, with the transcript as a real inner scroller (`.chat-transcript`, `overflow-y: auto`) and the body hard-locked (`html.chat-scroll-lock`). The composer and pills are overlays within the column; content scrolls _under_ the translucent header/composer (the under-glass look) via the scroller's top/bottom padding. The body-lock is what makes this safe: with no scrollable document behind the inner scroller, the mobile-WebKit touch-routing fight ADR-039 fled cannot arise (it needed a document to drag). It also kills the keyboard-up composer stutter that document scroll couldn't avoid — the column is sized by layout, so the composer never chases the keyboard frame-by-frame. **Do not reintroduce an inner/overlay chat scroller without the body-lock**, and do not switch chat back to document scroll (the stutter returns).
- **Header nav — one unified priority+ mechanism at every width.** Links live in `.header-right` and overflow **one at a time** into a `More` menu as the header narrows; on a phone every link ends up in `More` (which reads as the hamburger), so there is no separate breakpoint that dumps the whole nav at once. The `More` button only mounts when the overflow bucket is non-empty. Sidebar collapses behind a sheet/drawer triggered from the chat header. Friday 👑 always at the top of the drawer.
  - **Bounce-free by construction.** An off-screen ghost copy of the links supplies intrinsic widths; the available-width signal is `.header-right`'s box width — the flex _leftover_ after the content-stable `header-left`, which does **not** change when a link moves between the visible row and `More`. That severs the classic priority+ feedback loop (hide a link → container widens → link reappears → oscillate), so the visible count is a pure, monotonic function of the measured width. See `services/dashboard/src/routes/+layout.svelte`.
- **Rail pages collapse the rail to a filter sheet.** Pages built on `RailShell` (e.g. Memory) render the desktop left-rail as a rounded `Filters (n)` trigger row on mobile that expands an anchored dropdown **sheet** (`role="dialog"`, Escape + focus-trap). The `main` pane renders once across the breakpoint, so a rotation/resize never remounts an in-progress inline edit. See `docs/ui-conventions.md` → _Responsive rail layout_.

## List pages

- Card layout on mobile, table/grid on desktop.
- Sentinel-paginated via `IntersectionObserver` — top + bottom sentinels load older / newer pages.
- Avoid full-array renders for anything that could grow unbounded (chat, sessions, mail, evolve proposals, logs).

## Forms

- One field per row.
- Inputs ≥ 44px tall (touch target minimum).
- Full-width buttons.
- Bottom-anchored CTA (primary action stays in thumb reach).
- Modals become full-screen sheets on mobile.

## Detail pages

- Full-screen on mobile, in-place pane on desktop. Same content, different framing.

## Touch targets

- All interactive targets ≥ 44×44 px. Includes sidebar agent rows, autocomplete entries, attachment thumbnails.

## Chat virtualization

- Hand-rolled in ~150 LOC using `IntersectionObserver` sentinels at top/bottom + DOM windowing. No React-flavored libraries; Svelte 5 runes + ResizeObserver throughout.
- Window the rendered range — keep ~200 messages in DOM; recycle as the user scrolls.
- Auto-pin-to-bottom only when the user is already at the bottom (track `isAtBottom`).
- Bottom-sentinel: when off-screen, show a "↓ new messages" button.

## Mobile autocomplete (the slash-command picker)

- Reference: Slack mobile.
- Tap-to-insert must preserve keyboard focus. Standard `click` blurs the input on mobile, which then dismisses the picker.
- Use `pointerdown` + `preventDefault` on the autocomplete entry. The keyboard stays up; the slash-completion lands in the input; the user keeps typing.

## Camera and gallery uploads

- `<input type="file" accept="image/*" capture="environment">` for "open camera" affordance.
- Standard `<input type="file" multiple>` for gallery picker.
- HEIC files are converted to PNG by the upload handler (ADR-007 + `attachments.ts`); the dashboard doesn't care about the source format.

## Soft keyboard — the visual-viewport-sized column (ADR-041)

The chat route is a `position: fixed` column (`.chat-viewport`) whose **height is `var(--vv-bottom-y)`** — the visual viewport's bottom edge in layout-viewport coordinates (`vv.offsetTop + vv.height`). The composer is an absolutely-positioned overlay at the column's bottom; the transcript is the inner scroller. Because the composer is placed by **layout** (column bottom), not by per-frame JS, opening the keyboard simply shrinks the column and the composer re-lays-out above the keyboard — **no scroll-thread chasing, so no stutter.** Keyboard closed, `--vv-bottom-y` is the visible height above the iOS bottom URL bar, so the composer clears that too. (This replaces ADR-040's per-bar `translateY` anchor + settle-probe machinery, which existed only because the bars were `position: fixed` against the layout viewport under document scroll.)

One tracker — `$lib/util/keyboard-inset.ts`, started from `+layout.svelte` — writes the geometry on `:root` **in every state** (not just while focused, so the column is always correctly sized):

- **`--vv-bottom-y`** = `vv.offsetTop + vv.height`: the column height (and the keyboard / URL-bar clearance line).
- **`--vv-top-y`** = `vv.offsetTop` and **`--kb-inset`** = `max(0, innerHeight − vv.height)` are still written (consumed by the header offset and as belt-and-braces), but most of their ADR-040 consumers are gone with the fixed-bar model.
- **`keyboard-open` class** (focus-gated): zeroes `--kb-safe-bottom` while typing (the keyboard covers the home-indicator zone).

Presence is focus-derived (`focusin`/`focusout`, debounced blur) **only for the class**; the geometry vars track the visual viewport continuously via `vv.resize` + `vv.scroll` + `window.resize`, rAF-coalesced and change-guarded. The hard-won iOS 26.5 facts that constrain the formula (`vv.offsetTop` is the honest pan source, not `pageTop`; the pan can legitimately exceed `innerHeight − vv.height` in the shrunk regime so it is never clamped) are recorded in ADR-040 and still apply to `--vv-bottom-y`.

With the inner-scroller column, the **Safari-tab "first-tap gap" ceiling from ADR-040 is moot for placement** — the composer is at the column bottom regardless of the keyboard chrome allowance; the column is simply as tall as iOS says is visible.

### Diagnostics: `?kbdebug`

Append `?kbdebug` to the chat URL to render a live HUD of the tracker inputs/outputs (`innerHeight`, `vv.height`, `vv.offsetTop`, `vv.pageTop`, `scrollY`, the written vars, the class). In dev builds, keyboard opens additionally post geometry snapshots to `/api/_diag/client-error` (dashboard JSONL log) — the side-band used to verify on-device behavior the LAN dev origin can't otherwise surface (the installed PWA needs HTTPS, so PWA verification happens post-release with this HUD). Keep it until WebKit's keyboard reporting stabilizes.

## Mobile Send/Stop: `mousedown` blur suppression

On iOS, `mousedown` on any non-input element causes the currently-focused textarea to blur before `click` fires. The Send and Stop buttons carry `onmousedown={(e) => e.preventDefault()}` to suppress that blur without blocking the synthesized `click`.

Both the Send and Stop buttons carry `onmousedown={(e) => e.preventDefault()}`. `preventDefault` on `mousedown` stops the browser from transferring focus away from the textarea, suppressing the blur, without blocking the subsequent `click` — iOS synthesizes `click` from `touchstart`/`touchend` independently of `mousedown`.
