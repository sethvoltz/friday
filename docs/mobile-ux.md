# Mobile UX

Friday is mobile-first, not mobile-responsive-as-an-afterthought. Every page is fully usable on a phone, including chat with attachments, agent inspection, and dashboard tables.

## Layout primitives

- **Fixed header** on all viewports. No bottom tab bar.
- **The document is the only scroller â€” chat included.** The chat transcript is an inert in-flow element that scrolls the document/window like every other page; there is no nested or `position: fixed` chat scroller. The pinned bars (header, sidebar, composer) are `position: fixed` overlays layered over the document scroll, not competing scrollers. A fixed full-viewport overlay scroller intermittently lost the touch-routing fight to the document on mobile WebKit, dragging the page as a chunk â€” see ADR-039; don't reintroduce one.
- **Header nav â€” one unified priority+ mechanism at every width.** Links live in `.header-right` and overflow **one at a time** into a `More` menu as the header narrows; on a phone every link ends up in `More` (which reads as the hamburger), so there is no separate breakpoint that dumps the whole nav at once. The `More` button only mounts when the overflow bucket is non-empty. Sidebar collapses behind a sheet/drawer triggered from the chat header. Friday đź‘‘ always at the top of the drawer.
  - **Bounce-free by construction.** An off-screen ghost copy of the links supplies intrinsic widths; the available-width signal is `.header-right`'s box width â€” the flex _leftover_ after the content-stable `header-left`, which does **not** change when a link moves between the visible row and `More`. That severs the classic priority+ feedback loop (hide a link â†’ container widens â†’ link reappears â†’ oscillate), so the visible count is a pure, monotonic function of the measured width. See `services/dashboard/src/routes/+layout.svelte`.

## Mobile autocomplete (the slash-command picker)

- Reference: Slack mobile.
- Tap-to-insert must preserve keyboard focus. Standard `click` blurs the input on mobile, which then dismisses the picker.
- Use `pointerdown` + `preventDefault` on the autocomplete entry. The keyboard stays up; the slash-completion lands in the input; the user keeps typing.

## iOS soft keyboard â€” keyboard-height tracking

iOS raises the soft keyboard asynchronously after a text field is focused. `position: fixed` elements use the **layout viewport** as their reference, which does not shrink when the keyboard opens â€” so a `bottom: 0` composer bar stays pinned to the bottom of the layout viewport and ends up hidden behind the keyboard.

Current implementation:

- **Viewport meta** (`app.html`): `interactive-widget=resizes-content` â€” iOS ignores this hint; Android Chrome gets a free keyboard-shrinks-layout fix with no code changes.
- **`--kb-h`** (set by focusin/focusout handlers in `+layout.svelte`): `max(0, window.innerHeight - vv.height)` measured inside a `requestAnimationFrame` on `focusin`, so the keyboard animation settles before the height is read. Cleared to `0px` on `focusout`.
- **`.chat-input-floating`** in `ChatShell.svelte` uses `bottom: calc(1rem + var(--kb-safe-bottom, 0px) + var(--kb-h, 0px))` to lift the composer above the keyboard. `.jump-to-bottom-wrap` bottom and `.chat-transcript` padding-bottom include `--kb-h` as well so the whole layout shifts up together.
- **Header**: `position: fixed; top: 1rem`. No JS translation needed â€” iOS automatically pins `position: fixed` elements against the visual viewport top, so the header stays in place as the keyboard opens.
- **`scrollIntoView` on focus** (`ChatInput.svelte`): fires after a 100ms delay so the keyboard has begun raising before we scroll. Uses `behavior: "instant"` â€” `behavior: "smooth"` is unreliable on WebKit (bug #238497).

`--kb-h` is driven by `document` `focusin` and `focusout` only â€” no `visualViewport` `resize` listener. `vv.resize` fires on iOS Safari whenever the URL bar collapses during scroll (not just on keyboard open/close), causing ~50px false keyboard heights mid-scroll and composer snapping. Restricting measurement to focus events eliminates that noise; the `requestAnimationFrame` defer on `focusin` lets the keyboard animation settle before the height is sampled.

## Mobile Send/Stop: `mousedown` blur suppression

On iOS, `mousedown` on any non-input element causes the currently-focused textarea to blur before `click` fires. For the Send and Stop buttons this means: tap â†’ textarea blurs â†’ the `focusout` handler resets `--kb-h` to 0px â†’ composer snaps down â†’ by the time `click` fires, the button has shifted and the event may misfire.

Both the Send and Stop buttons carry `onmousedown={(e) => e.preventDefault()}`. `preventDefault` on `mousedown` stops the browser from transferring focus away from the textarea, suppressing the blur, without blocking the subsequent `click` â€” iOS synthesizes `click` from `touchstart`/`touchend` independently of `mousedown`.
